```bash
ctxsift doctor
```

Run this after `ctxsift configure`. Doctor checks every part of the runtime that CtxSift depends on and tells you what is working, what is missing, and what is optional.

<Aside type="tip">
Doctor does not modify anything. It is safe to run at any time — in CI, after changing config, or after upgrading.
</Aside>

---

## What doctor checks

Doctor runs all checks in one pass and reports each one with a severity and status. The checks are grouped below by what they cover.

### Required checks

These must pass for CtxSift to function at all.

| Check | What it tests |
|---|---|
| `database` | Can CtxSift create and initialize its SQLite workspace database at the resolved path |
| `sqlite_core` | Core SQLite is importable and connectable |
| `sqlite_fts5` | SQLite FTS5 virtual table extension is available (used for lexical recall search) |

If `database` fails, the resolved DB path is printed in the error. Common causes: a path that does not exist yet, or a permissions issue on the target directory.

If `sqlite_fts5` fails, your Python's SQLite was built without FTS5 support. This is rare on standard installs but can happen with some minimal system builds.

### Warning checks

These are not hard blockers, but failures mean reduced functionality or a misconfiguration that will cause problems when you actually run a command.

| Check | What it tests | When it fires |
|---|---|---|
| `git_workspace` | Whether the current directory is inside a Git repo | Always — non-Git workspaces get reduced freshness metadata |
| `remote_config` | Whether `remote.base_url` is set but `remote.model_name` is missing | Only when remote mode is enabled |
| `sqlite_vec` | Whether the `sqlite-vec` extension loaded and whether the indexed model matches the configured embedding model | Always — mismatch means recall degrades to FTS5 only |
| `local_runtime` | Whether the configured local model path is resolvable as either llama.cpp or Transformers | Only when local compression is active |
| `litellm` | Whether LiteLLM is installed | Only when `remote.base_url` is set |
| `gguf_artifact` | Whether the configured `.gguf` file is already in the local Hugging Face cache | Only on CPU llama.cpp path |

### Optional checks

These affect performance and feature availability but do not block operation.

| Check | What it tests | Impact if missing |
|---|---|---|
| `cuda` | Whether PyTorch sees a usable CUDA device | Local GPU compression and GPU embedding acceleration unavailable |
| `onnxruntime` | Whether ONNX Runtime is installed | Embedding inference stays on the Torch path (slower on CPU) |
| `flashattention` | Whether `flash-attn` or a compatible kernels package is installed | Attention defaults to SDPA; throughput may be lower on supported GPUs |
| `llama_cpp` | Whether `llama-cpp-python` is installed | CPU llama.cpp path unavailable |
| `accelerate` | Whether Hugging Face Accelerate is installed | Required for BitsAndBytes quantization on CUDA |
| `bitsandbytes` | Whether BitsAndBytes is installed | Required for `bnb-8bit` / `bnb-4bit-*` quantization modes |

## Reading the output

Each check is printed as:

```
[severity] check_name: status - detail
```

For example:

```
[required] database: ok - Database initialized at /repo/.git/ctxsift/ctxsift.db (schema=v3).
[warning]  sqlite_vec: ok - sqlite-vec is available (vec_version=0.1.3, dim=896).
[optional] cuda: ok - CUDA is available.
[optional] onnxruntime: warning - ONNX Runtime is not installed.
[warning]  gguf_artifact: ok - Configured GGUF artifact ibm-granite/granite-4.0-350m-GGUF/granite-4.0-350m-Q8_0.gguf is not cached locally yet. Configure or first use will download it.
```

A `warning` or `optional` status on a check does not mean doctor failed — it means that specific feature is reduced or the configuration has an issue worth looking at. Only `required` failures block basic operation.

## Common failures and fixes

**`database` fails — path error**

CtxSift could not initialize the SQLite database. Usually a missing parent directory or a permissions issue.

```bash
# Check what path CtxSift is trying to use
ctxsift config show | grep db_path
```

**`sqlite_fts5` fails**

Your Python build's SQLite does not include FTS5. Try a standard Python installer from [python.org](https://www.python.org/downloads/) rather than a minimal system package.

**`remote_config` fails — missing model name**

`remote.base_url` is set but `remote.model_name` is empty. Fix:

```bash
ctxsift config set remote.model_name gpt-4o-mini --global
```

**`sqlite_vec` — model mismatch warning**

The vector index in the database was built with a different embedding model than what is currently configured. Recall will degrade to FTS5 only. Either revert to the previous model or reinitialize the workspace database to rebuild the index with the new model.

**`litellm` fails — not installed**

Remote compression is configured but LiteLLM is missing:

```bash
uv tool install "ctxsift[remote]"
```

**`llama_cpp` fails — not installed**

CPU local compression path requires `llama-cpp-python`, which is bundled in the base install. If it is missing, reinstall:

```bash
uv tool install ctxsift --force-reinstall
```

**`gguf_artifact` — not cached yet**

Not a failure. The GGUF file will be downloaded automatically on first compress run. Doctor is just telling you it is not pre-cached yet.

**`cuda` optional — unavailable**

PyTorch is not installed or does not see a GPU. If you want GPU acceleration:

```bash
uv tool install "ctxsift[gpu]"
```

**`onnxruntime` optional — not installed**

Embedding inference on CPU stays on the Torch path. ONNX is faster for the default Harrier model on CPU. It is installed automatically with the base package on most setups. If it is missing:

```bash
uv tool install ctxsift --force-reinstall
```

**`flashattention` optional — not installed**

Flash Attention is not required. SDPA is the fallback. On Windows, Flash Attention is usually not available and this is expected.

## Typical setup flow

```bash
ctxsift configure   # set model and config
ctxsift doctor      # verify everything resolves
```

If doctor shows no `required` failures, you are ready. Optional and warning checks are worth reading but do not block use.
